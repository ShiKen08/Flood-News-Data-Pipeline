Flood News Data Pipeline

Technical documentation & development history — generated 2026-05-21

This pipeline retrieves, cleans, and classifies flood-event news articles from the Common Crawl open web archive. Starting from a structured list of historical flood events (date, location, country), it:

  1. Queries the Common Crawl CDX index for URLs archived around each event's time window
  2. Downloads raw WARC byte-range slices from Common Crawl S3 using HTTP Range requests
  3. Extracts article body text with trafilatura ML extraction + BeautifulSoup fallback
  4. Scores keyword relevance and filters junk pages in a fully vectorised NLP pass
  5. Runs a fine-tuned SetFit multilingual classifier to identify genuine event-reporting articles
  6. Applies a 17-rule post-filter to systematically remove false positives by category
  7. Performs a keyword-confirmation audit to remove domain-sweep contamination

Scope: 269 flood events across the Americas, 2015–2025, in four languages: English, Spanish, Portuguese, and French. Designed to run on a SLURM HPC cluster (Snellius, SURF) in parallelised batch mode.

Each stage reads parquet files produced by the previous stage and writes its own parquet to output/. This means every stage is independently re-runnable without reprocessing upstream stages. On the SLURM cluster, events are split into batches; each SLURM job runs stages 01–09 on its slice and writes to output/batch_{start}_{end}/. After all jobs finish, scripts/merge_batch_outputs.py merges everything into the main output/ directory, deduplicating on natural keys.

Stage Documentation

Stage 00 — Pre-Flight Setup  stage_00_preflight.py

Initialises the pipeline by fetching Common Crawl collection metadata, computing time windows for each flood event, assigning query languages per event, and building a location dictionary of normalised place names and aliases. Runs once before batch processing begins and is skipped on reruns if outputs already exist.

Inputs

  • data/flood_crawl.csv — 269 events with dates, locations, ISO codes
  • collinfo.json — CC crawl index (fetched from API, cached locally)

Outputs

  • crawl_coverage.parquet — per-event CC window status and matching crawl IDs
  • language_assignments.parquet — per-event query language code lists
  • location_dictionary.parquet — normalised place names, aliases, ambiguity flags

Time Window Computation

For each flood event, a query window of 5 days before the event start and 45 days after the event end is computed (capped at 21 post-event days for long-duration events >20 days). This window is intersected against Common Crawl crawl dates. If no crawl falls within the window, the nearest post-window crawl within 120 days is used as a fallback (NEAREST_CRAWL status) rather than discarding the event entirely.

Why 45 days post-event? Common Crawl is not real-time — a page published on the flood date may not be archived until the next crawl cycle, which can be weeks later. Extending to 45 days captures delayed-indexing follow-up coverage that would otherwise be missed.

Language Assignment (Three-Tier System)

Languages are assigned in tiers based on Common Crawl coverage breadth. Tier 3 languages (20 low-coverage languages) automatically fall back to French and English. Every event always also receives universal fallback languages (English, Spanish, Portuguese) regardless of the event's native language, to capture international press coverage.

Location Dictionary

Place names are parsed from the Location and River Basin fields. Administrative suffixes and directional prefixes are stripped to yield the core place name. Parenthetical content is extracted as additional alias terms. A known-ambiguous name list (georgia, columbia, victoria, santa cruz, lima, etc.) flags places that require a co-occurring flood keyword in the same sentence before being counted as a location hit in Stage 06.

Why flag ambiguous names? "Santa Cruz" alone matches Santa Cruz, California; Santa Cruz, Bolivia; and several others. Requiring a flood term in the same sentence greatly reduces false geographic matches without losing genuine hits.

Stage 01 — Query Specification Building  stage_01_query_specs.py

Combines language codes, location terms, and flood keyword lexicons into structured CDX query specifications. Each specification defines exactly what URL pattern to search for, on which domains, and with which retrieval strategy.

Inputs

  • crawl_coverage.parquet
  • language_assignments.parquet
  • location_dictionary.parquet
  • data/keyword_lexicon.json — flood terms per language
  • data/source_domain_list.json — per-country curated domains

Outputs

  • event_query_specs.parquet — one row per (flood, crawl, variant, domain)

Query Variants

VariantContentStrategy
AFlood keywords onlyBroad recall, many false positives
BLocation names + aliases onlyLocation-anchored, topic-agnostic
C (primary)Flood keywords AND locationDefault — precision-optimised
DImpact/emergency terms AND locationOptional secondary for impact quantification
Why Variant C only in production? Variant A returns too many unrelated articles (any page mentioning "flood" anywhere). Variant C constrains the query to URLs containing both flood vocabulary and the specific event location, producing far higher initial precision without significant recall loss for event-specific reporting.

Domain Strategy

Restricted: queries the country's curated domain list from source_domain_list.json. Open-web fallback: for countries without a domain list, or as a secondary pass, queries international humanitarian sites: reliefweb.int, floodlist.com, ifrc.org, unocha.org, and others. The open-web strategy never uses ccTLD wildcards as these cause 504 timeouts on the CC CDX API.

Stage 02 — Common Crawl Index Query  stage_02_query_cc_index.py

Queries the Common Crawl CDX (Columnar Data Exchange) index to retrieve URL pointers matching each query specification. Handles pagination, rate limiting, per-domain hit capping, and resumable caching of raw responses.

Inputs

  • event_query_specs.parquet
  • data/source_domain_list.json

Outputs

  • cc_index_hits.parquet — raw URL pointers (url, filename, offset, length, digest)
  • raw_index_responses/ — JSONL cache per (flood, crawl, domain) for resumability
  • hit_count_summary.parquet — counts per (flood, query, crawl, domain)

Per-Domain Hit Caps

  • 3 000 hits per domain per flood — default cap
  • 500 hits for national/international scope domains
  • Open-web humanitarian fallback domains are always exempt from the national cap
Why cap hits? High-traffic outlets return enormous numbers of results for broad flood queries. Without caps, El Tiempo (Colombia) alone returned 8 000+ hits for "Colombia inundación" — mostly archive pages, opinion columns, and unrelated articles. The cap enforces source diversity and keeps download volume manageable.

Error Handling Strategy

HTTP StatusAction
404Domain not in this crawl — silent skip, no retry
400 / 504Bad query pattern or too broad — skip immediately, no retry
Empty 200 bodyCC silent rate-limiting signal — sleep 10 s, retry up to 3 times
Other HTTP errorsExponential backoff (2^attempt seconds), max 3 retries

Parallel Execution & Resumability

Queries run in a ThreadPoolExecutor (8 workers), giving ~10× throughput vs sequential. Raw CC responses are saved as JSONL files per (flood, crawl, domain) immediately after receipt. On resume, any combination with an existing JSONL file is skipped — the cache is re-read instead of re-queried.

Stage 03 — Pointer Validation & Deduplication  stage_03_validate_pointers.py

Validates the structural integrity of CC pointers returned by Stage 02, filters by byte size, and deduplicates at multiple levels to maximise download efficiency.

Inputs

  • cc_index_hits.parquet

Outputs

  • validated_pointers.parquet — valid pointers with dedup flags
  • rejects.parquet — invalid/undersized pointers with reasons

Validation Rules

A pointer is rejected if: offset is null or negative; length is null or zero; filename does not match the WARC path pattern.

Size Filter

Pointers with length < 500 bytes → rejected as TOO_SMALL. Pointers exceeding 5 MB → flagged as TOO_LARGE but retained.

Deduplication Layers (applied in order)

  1. Exact pointer dedup within flood_id: same (filename, offset, length) → keep one copy
  2. Content digest dedup within flood_id: same CC digest hash → keep first
  3. URL dedup within flood_id: same URL archived multiple times → keep most recent timestamp
  4. Cross-event flag (kept, not dropped): URLs appearing in multiple flood_id groups flagged with cross_event_shared=TRUE

Stage 04 — WARC Download  stage_04_download_warc.py

Downloads WARC byte-range slices from Common Crawl S3 using HTTP Range headers. Uses asyncio for 50 concurrent downloads with two-phase timeout handling designed for Common Crawl's characteristic trickle-rate delivery.

Inputs

  • validated_pointers.parquet

Outputs

  • cache/{flood_id}/{pointer_id}.warc.gz — cached WARC slices
  • warc_fetch_log.parquet — per-pointer download result

Two-Phase Download Strategy

Primary pass (30 s timeout, 10 KB/s floor): All pointers attempted concurrently (50 asyncio coroutines). Any download slower than 10 KB/s sustained is cancelled at the event-loop level.

Deferred pass (300 s timeout): Pointers that timed out in the primary pass retried with a generous 5-minute timeout and no throughput floor.

Why asyncio instead of threading? Common Crawl throttles connections to ~1 KB/s per stream. With requests + stream=True, a slow download holds a Python thread indefinitely. asyncio.wait_for() cancels at the event-loop level regardless of trickle delivery — this was the only reliable way to enforce timeouts on CC downloads.

Retry & Rate Limit Handling

  • Max 3 retries per pointer with exponential backoff: 5 s, 25 s, 125 s
  • 5 consecutive 403/503 responses → 120 s rate-limit pause
  • 20 consecutive primary-pass timeouts → 60 s burst-throttle pause
  • Checkpoint every 500 completed downloads
  • Resume detection: cached files < 95% of expected length are re-downloaded

Stage 05 — Text Extraction  stage_05_extract_text.py

Extracts article body text from cached WARC records using a two-tier approach: trafilatura ML extraction first (better quality), BeautifulSoup fallback second (higher coverage).

Inputs

  • warc_fetch_log.parquet
  • cache/{flood_id}/{pointer_id}.warc.gz

Outputs

  • extracted_text.parquet — doc_id, page_title, meta_description, raw_text, extraction_method, encoding_detected

Extraction Pipeline

  1. Parse WARC record headers; skip non-HTTP-response records
  2. Skip non-HTML responses (PDFs, JSON, binary content)
  3. Try trafilatura: ML model that identifies main article body and removes sidebars, navigation, cookie banners, and footers
  4. If trafilatura returns nothing: fallback to BeautifulSoup full-body extraction
  5. Extract page_title and meta_description from HTML <head>
  6. Detect character encoding with chardet
Why prioritise trafilatura? News sites include extensive sidebars, "related articles" sections, and breaking-news tickers. Without ML-based extraction, a sports article might score high on flood keywords because a flood headline appears in a sidebar ticker. Trafilatura isolates the main article body, dramatically reducing false positive keyword scores.

Stage 06 — Clean, Detect & Score  stage_06v_clean_deduplicate.py

The central quality gate. Runs 7 sequential processing steps: pre-filtering junk URLs, cleaning text, checking usability, detecting language, scoring flood relevance, filtering by publication date, and deduplicating content.

Inputs

  • extracted_text.parquet
  • validated_pointers.parquet
  • event_query_specs.parquet
  • language_assignments.parquet
  • location_dictionary.parquet
  • data/keyword_lexicon.json

Outputs

  • clean_text.parquet — all scored documents with:
    is_relevant, is_soft_relevant, is_event_article,
    flood_term_hits, location_term_hits, subnational_hits,
    pub_in_window, is_content_duplicate, language_detected
  • rejects.parquet

Relevance Scoring

Per flood_id group, each document receives four scores: flood_term_hits (keyword vocabulary), location_term_hits (event location matches), subnational_hits (sub-country level — required for is_relevant=TRUE), and location_specificity_score.

Why require subnational hits specifically? A document mentioning "Colombia" and "inundación" could be a general article about Colombia's chronic flooding history, not a report on a specific event in Chocó Department. Requiring a match to the specific affected district/province ensures event specificity.

A fast substring pre-filter runs before regex: documents with no flood-term substring anywhere bypass the compiled regex entirely. Since >95% of crawled documents contain no flood vocabulary at all, this gives roughly 10× speedup on Stage 06.

False Positive Taxonomy

The SetFit ML classifier achieves strong recall but generates systematic false positives in several predictable categories. Understanding these categories explains the post-filter design and guides future improvements.

Why the classifier over-fires

SetFit is trained on event-reporting articles — short, present-tense, location-specific texts with urgency language and impact vocabulary. During inference, it encounters articles that match this structural profile but are about other types of events or disasters. The multilingual model also sometimes confuses Spanish and Portuguese crisis-language registers across disaster types.

Category 1 — Strict-Domain False Positives (134 rejections)

Certain domains produce high rates of non-event articles that match the vocabulary pattern. ElTiempo.com (Colombia) publishes long-form archive journalism — any article on Colombia's chronic flood season passes keyword and model filters but is not event-specific. The strict-domain rule requires an explicit flood term in the article title before passing these sources.

Category 2 — Weather Forecast Pages (88 rejections)

Meteorological agency websites contain flood vocabulary (rainfall amounts, river level warnings, flood watches) but are forecast outlooks, not event reports. Filtered by title patterns matching "Weather | Forecast Conditions" and Spanish/Portuguese forecast language.

Category 3 — COVID Content (38 rejections)

During 2020–2022, many articles discuss both COVID-19 and natural disaster emergency management simultaneously. Filtered when COVID/pandemic terms appear in the title without accompanying flood terms.

Category 4 — Earthquake Articles (14 rejections)

Earthquake articles slip through because news site pages include related-articles carousels and breaking-news tickers containing flood headlines from a different story. Filtered when earthquake/seismic terms appear in the title without flood terms.

Category 5 — Other Disaster and Off-Topic Content (38 rejections combined)

Wildfire, disease outbreak, crime/narco, and mine disaster articles all generate false positives when they occur in the same geographic region and time period as a flood event. In each case, the post-filter fires only when the off-topic term is present in the title and no flood term is also present in the title.

Category 6 — Institutional and Non-Article Pages (76 rejections combined)

Press release portals, council meeting pages, staff profile pages, CEMADEN procurement pages, and category archive indexes all contain enough flood vocabulary to score positively. Filtered by structural title patterns and URL path patterns.

Category 7 — Domain-Sweep Contamination (2 318 articles removed in post-audit)

The most significant category discovered during the May 2026 dataset quality audit. Common Crawl periodically archives an outlet's entire front-page and navigation structure in a single crawl pass, not just the article retrieved for the specific flood event. This creates domain-sweep contamination: a query for flood 11 (Oriximiná/Pará, 2021) against brasil.elpais.com returns not just flood articles but the entire recent article archive of El País Brasil — including Bolsonaro politics, football, crime, and celebrity news.

Systematic sampling confirmed: flood 11 (1 523 articles) had 91% junk across three independent 500-article samples — removed entirely. Flood 79 (1 198 articles) had 75% junk — 894 articles removed, 304 keyword-confirmed kept. After keyword-filtering all Spanish and Portuguese articles across all flood IDs, an additional 384 junk articles were removed. Total removed in post-audit: 2 318 articles.

Mitigation: the keyword-confirmation filter is now the final step in the pipeline. Any article passing the ML classifier and post-filter must also contain at least one flood vocabulary term in either the title or the first 300 characters of body text before being included in the final dataset.

Remaining Precision Estimate

Starting from ~35–40% raw precision on model positives, the post-filter removes systematic false positives, and the keyword-confirmation audit removes domain-sweep contamination, yielding an estimated 85–90% precision on the final 612 verified articles. The primary remaining sources of false positives are: articles that discuss a flood in passing within a broader policy context, and opinion/analysis articles that use event-specific location vocabulary without being primary event reports.
